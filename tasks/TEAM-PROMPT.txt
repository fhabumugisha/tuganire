TEAM EXECUTION PLAN

This document orchestrates parallel implementation of the Tuganire MVP using Claude Code Agent Teams. A coordinator agent reads this file and dispatches work to implementer agents. The coordinator NEVER writes code — it assigns, monitors, verifies and unblocks.

PROJECT SUMMARY

Tuganire — a French to Kinyarwanda conversational voice-translation app (Spring Boot 4 backend with an LLM + native-correction layer + STT/TTS, a Python MMS-TTS server, a Thymeleaf/HTMX web POC, and a Kotlin/Compose Android app).

DELIVERY ORDER (web-first, decided by the project owner): build and validate the WEB version first (Phase 1 = Waves 1-8 below, tasks 01,02,04-20,22-24), THEN build Android (Phase 2 = tasks 03 and 21) only after the web POC is validated. The coordinator MUST NOT spawn tasks 03 or 21 during Phase 1.

Phase 1: 22 tasks across 8 execution waves. Peak concurrency: 6 agents (Wave 4). Recommended team size: 6 agents.

AGENT SPAWN TEMPLATE

For each agent, the coordinator spawns via the Agent tool:

  subagent_type: implementer
  mode: bypassPermissions
  name: impl-{task-area}
  prompt: |
    You are impl-{task-area} in the Tuganire implementation team.

    Read and implement the task file at tasks/{NN-task-name}.txt.
    Follow ALL instructions exactly. Respect PROJECT CONTEXT, ARCHITECTURE and
    TECH STACK AND BEST PRACTICES. Only create/modify the files listed in
    WHAT TO IMPLEMENT. Run ./mvnw spotless:apply then ./mvnw clean compile
    (backend tasks) and verify every item in ACCEPTANCE CRITERIA before
    reporting done. Report completion via SendMessage when finished; if blocked,
    report the blocker via SendMessage immediately.
    For tasks 20 and 23: use the ui-ux-pro-max skill for page design, the
    frontend-design skill for HTML development, and review the result against
    rules/thymeleaf-rules.mdc before declaring done.

COORDINATION RULES

1. The coordinator NEVER implements code. Only assigns, monitors, and unblocks.
2. Spawn all tasks in a wave simultaneously (single message, multiple Agent calls).
3. When a wave completes, run the wave checkpoint (below), then spawn the next wave.
4. If an agent reports a blocker, analyze it and send guidance via SendMessage.
5. If an agent fails, reassign the task to a new agent with extra context about the failure.
6. Track progress per task: pending, in-progress, complete, failed.
7. After all waves complete, run final validation: ./mvnw -B -ntp clean verify (backend), npm run build-prod (CSS), ./gradlew :app:assembleDebug (android), and the Python server health check.

WAVE CHECKPOINTS (run before advancing)

- After Wave 1: backend ./mvnw clean compile green; Python server starts + /health 200; Android assembleDebug green.
- After Waves 2-6 (backend): ./mvnw clean compile green (and ./mvnw -B -ntp clean verify after Wave 4 once Redis/Postgres are up via docker-compose).
- After Wave 7: ./mvnw -B -ntp clean verify green (Testcontainers ITs); web POC serves GET /.
- After Wave 8: full final validation; README + CI present.
